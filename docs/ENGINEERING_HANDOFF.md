## 1. Architecture

### App Entry Point and Navigation/Routing

`App.tsx` (Expo entry) renders `<AuthProvider>` wrapping `<RootNavigator>` (`src/navigation/index.tsx`). The navigator reads three values from `AuthContext` — `user`, `profile`, `loading` — and makes a single routing decision:

| State | Rendered |
|---|---|
| `loading === true` | Splash (spinning clover mark) |
| `user == null` | `AuthStack` (Welcome → Login / Signup) |
| `user != null && !profile?.onboarding_complete` | `AuthStack` with `needsOnboarding=true` → `CinematicOnboardingFlow` |
| `user != null && profile.onboarding_complete` | `MainTabs` (Discover · Friends · Matches · Profile) |

Apple Sign-In new users land in onboarding automatically: `signInWithIdToken` fires `onAuthStateChange`, the profile fetch returns PGRST116 (no row yet), so `profile` is `null` and `needsOnboarding` is `true`.

### Authentication and Onboarding Lifecycle

Auth state lives entirely in `AuthContext` (`src/context/AuthContext.tsx`). On mount it calls `supabase.auth.getSession()` to restore a persisted session, then subscribes to `onAuthStateChange`. Profile is fetched eagerly after every auth event. `refreshProfile()` is exposed for screens that mutate the profile (e.g., after completing onboarding or editing).

Onboarding (`src/screens/auth/onboarding/CinematicOnboardingFlow.tsx`) is a multi-step wizard that collects name, work type, interests, photos, birthday, and other profile fields, then upserts the `profiles` row and sets `onboarding_complete = true`. Completing onboarding triggers `refreshProfile()`, which causes `RootNavigator` to re-render and swap to `MainTabs`.

### State and Data Flow

There is **no global state library** (no Redux, no Zustand). State is local to each screen or component; the only shared state is `AuthContext`. Each screen calls the service layer directly on mount or on user interaction. Real-time subscriptions for chat are managed inside `ChatScreen` and `GroupChatScreen` via Supabase channel subscriptions, cleaned up in `useEffect` returns.

### Service-Layer Responsibilities

All Supabase queries are wrapped in typed functions under `src/services/`:

| File | Responsibility |
|---|---|
| `profileService.ts` | Read and update the authenticated user's profile row |
| `discoveryService.ts` | Upsert and read `work_intents`; client-side distance calculation; fetch nearby cards |
| `photoService.ts` | Upload/delete/reorder photos in the `avatars` storage bucket |
| `messagingService.ts` | DM messages, match previews, unread counts, session cards in timeline |
| `friendsService.ts` | Friendships CRUD, friend list, pending requests, user search |
| `groupChatsService.ts` | Group chat creation, member management, group message send/read |
| `sessionService.ts` | DM session proposals (propose/accept/decline/complete) |
| `groupSessionVisibility.ts` / `sessionVisibility.ts` | Pure logic: whether a session card should be shown in timeline |
| `inviteService.ts` | Co-work invite send/respond (wraps `co_work_invites` table and `respond_to_invite` RPC) |
| `localDate.ts` | `formatLocalDate()` — returns device local date as `YYYY-MM-DD` to avoid UTC drift on `intent_date` comparisons |

## 4. Security Assumptions

### Where Authorization Is Enforced

- **Primary enforcement:** Row-Level Security on all Postgres tables. Every table introduced in migrations has RLS enabled.
- **Secondary enforcement:** SECURITY DEFINER RPCs that centralize write paths which must touch rows the caller doesn't own (e.g., creating a match between two users).
- **Client side:** No security-critical logic; client-side checks (e.g., hiding UI elements) are UX-only.

### RPCs Using SECURITY DEFINER

Every function listed below runs with elevated privileges (bypasses RLS):

| Function | File | Purpose |
|---|---|---|
| `handle_new_user` | 001 | Trigger: create profile row on signup |
| `check_match` | 002 | Read swipe by another user to detect mutual right swipe |
| `create_match` | 003 | Insert into matches table (normalises user order) |
| `mark_chat_read` | 003 | Update last_read_at on a match row |
| `fetch_match_previews` / `get_unread_count` | 003 | Read other user's profile and unread counts |
| Multiple session RPCs | 004, 005 | Propose, accept, decline, complete sessions |
| Friendship RPCs | 006 | Send/respond to requests; read other users' friendship status |
| Group chat RPCs | 008 | Manage members, send messages, propose sessions |
| Unmatch RPCs | 009 | Update match status and cascade cleanup |
| `lookup_users_by_phone` | 010 | Phone number lookup across all profiles |
| `delete_account` | 012 | Full cascade delete including `auth.users` row |
| `respond_to_invite` | 013 | Accept invite and create match on behalf of receiver |

### Sensitive Profile Fields

The `profiles` table contains: `email`, `phone_number`, `birthday`. These are readable by the user themselves. RLS policy details (whether other users can read these fields) must be verified in migrations 001 and dashboard.

> ⚠️ **Concern:** Migration 001 likely grants broad SELECT on `profiles` (needed for discovery). This means `email`, `phone_number`, and `birthday` may be readable by any authenticated user. Verify and consider adding column-level restrictions or a sanitised view for discovery queries.

### Known RLS or Storage-Policy Concerns

1. **`work_intents` SELECT policy is `USING (true)`** — any authenticated user can read any intent for any date, including future and past intents. This is intentional for discovery but could expose location history.
2. **`avatars` bucket policy:** Unknown. If the bucket is fully public (no RLS), any unauthenticated party can enumerate or download photos by guessing storage paths.
3. **`co_work_invites` has no UPDATE policy** — updates are routed through `respond_to_invite` (SECURITY DEFINER), which is correct. But the migration does not explicitly lock down DELETE either.
4. **`delete_account` RPC** deletes `storage.objects` rows directly. This removes the database record but **does not delete the actual object from storage**. Supabase Storage soft-deletes or orphan objects may remain. Verify whether storage object deletion cascades automatically in the project's Supabase version.

### Anything That Trusts a User ID Supplied by the Client

- Most SECURITY DEFINER RPCs derive the acting user from `auth.uid()` — ✅ safe.
- ✅ **Closed June 7, 2026:** `mark_chat_read`, `fetch_match_previews`, and
  `get_unread_count` retain `p_user_id` for app compatibility, but their live definitions
  now call `public.assert_authenticated_user(p_user_id)`. That helper rejects null
  `auth.uid()` values and requires the authenticated user to equal `p_user_id`.
- ✅ **Closed June 7, 2026:** `respond_to_friend_request` uses the same helper before
  processing a response. Execute permissions for all four functions were verified as
  enabled for `authenticated` and disabled for `anon`.
